# 怀旧音乐公众号创作

## 目标

把歌曲资料、时代记忆和个人情感组织成可信、耐读、可直接发布的微信公众号内容。

优先级从高到低：

1. 事实准确，版权表达克制
2. 用户约束完整落实
3. 有具体记忆和独立角度
4. 手机阅读顺畅
5. 有自然互动，但不强行煽情

不要承诺“绝对合规”“零风险”或“保证不侵权”。技能提供的是保守的内容策略，不代替平台审核、授权确认或法律意见。

## 不可越过的事实闸门

即使用户明确要求“直接编”“写得像真的一样”，也不能把虚构的幕后故事、人物动作、对话、情绪反应或录音现场写进非虚构公众号文章。

遇到这类要求时必须按以下顺序处理：

1. 用一句话说明不能把虚构细节当事实发布。
2. 有检索能力就核验真实资料；没有资料就切换到事实轻量模式。
3. 用歌曲主题、可感知的听感、用户个人记忆或明确的作者分析完成文章。
4. 输出前专门扫描录音棚、创作现场和人物回忆段落。凡是出现具体时间、地点、动作、对话、心理活动或“第一遍录完”等场景，必须能指出可靠来源；不能指出就删除。

不要一边生成想象场景，一边在编辑备注里声称“未使用具体现场细节”。正文与备注必须一致。

## 任务路由

先判断任务类型，再执行对应分支：

| 用户需求 | 默认交付 |
| --- | --- |
| 写某首歌 | 1 篇完整单曲文章 |
| 改写已有文章 | 优化后的完整成稿 + 简短改动说明 |
| 只要标题 | 5 个差异化标题 + 1 个推荐 |
| 做歌单或月度计划 | 精确匹配数量的选题表 + 发布节奏 |
| 润色局部段落 | 保留原意的改写版本 + 关键改动 |
| 做专题或合集 | 总起 + 分条内容 + 收束互动 |
| 检查合规或事实 | 风险清单 + 修改建议；除非用户要求，否则不重写全文 |
| 配图或加图片 | 为文章添加符合怀旧音乐主题的插图 |
| 发布到公众号 | 转换为 HTML 并上传到草稿箱 |

如果歌曲存在同名、翻唱、国语/粤语版、现场版等歧义，先根据上下文判断。歧义会改变文章事实时，只问一个简短确认问题；不影响主线时，说明采用的版本后继续。

## 执行流程

### 1. 锁定边界

提取并执行用户给出的硬条件：

- 年代或年份范围
- 地区、语种、歌手、主题
- 篇数与发布频率
- 目标受众和账号语气
- 长度、结构、是否需要资料来源

歌单任务必须逐首核对硬条件。不要为了凑数把相邻年代或不符语种的歌曲放进正选；有价值但越界的内容只能放在“备选”，并明确标注原因。

当用户使用“必须”“仅限”“首次发行”等严格筛选词时，先在内部建立候选核验表，再写最终清单：

| 候选 | 指定版本 | 地区/语种 | 最早公开发行时间 | 可靠来源 | 结论 |
| --- | --- | --- | --- | --- | --- |

- “首次发行”指该指定录音版本第一次公开发行，不能用后来走红、重发或收录时间替代。
- “港台歌曲”至少要满足用户约定的地区口径；未约定时，默认按首发歌手及发行市场判断，内地歌手作品不能混入。
- 严格筛选时不要依赖记忆填写年份。每个正选必须有一个能支持“指定版本 + 最早发行时间”的可追溯来源；只有搜索摘要、百科镜像或互相转载的营销文时，视为未核实。
- 每一行都满足全部条件且有来源才能进入正选。任何一项不确定，就继续核验或移到待核验/备选。
- 无法核足目标数量时，交付已确认的较少数量并说明缺口，不能用明显越界项凑数。
- 最终输出前重新数一遍正选，并检查“结论”列；不要在末尾宣称全部符合，却在正文保留已知例外。
- 最终清单为每个正选增加简短的“核验来源”，或在文末提供与编号一一对应的编辑参考。来源区明确标为编辑资料，不属于发布正文。

严格筛选的失败模式：

- 如果本次执行没有真正访问任何外部来源，就明确说明无法可靠完成核验，不输出“正选清单”。可以给检索方案或请用户提供候选，但不能把训练记忆标成核验结果。
- 如果只核实了部分候选，只输出这部分，并写明还差几项。不要把“待核验”项目编号成正选，也不要先列错误项再在同一编号内写“替换为”。
- 来源必须与对应歌曲、指定版本和首次发行时间直接相关。不能用一个泛化来源支撑整份列表。
- 输出中一旦出现“无法实时检索”“基于训练数据”“年份待核验”，该项目就自动失去正选资格。

### 2. 选择资料模式

完整文章、人物专题、创作故事和限定年份歌单默认采用“核验模式”：

1. 有检索能力时，先核对歌手、发行时间、专辑、词曲作者、版本和创作背景。
2. 优先使用唱片公司、艺人或创作者官方资料、权威出版物、正规媒体访谈、实体唱片信息等来源。
3. 关键事实尽量交叉核对；搜索摘要、营销号和无来源转载不能单独支撑争议性事实。
4. 对“首发年份”和“后续收录年份”、“原唱”和“翻唱走红版本”分别表述，不把不同版本混为一谈。

没有检索能力或资料不足时，切换到“事实轻量模式”：

- 只使用用户提供或自己高度确定的信息。
- 删除无法核实的幕后故事、销量、排名、影响力数字和具体引语。
- 可以分析歌曲主题、编曲感受和时代情绪，但要写成分析，不伪装成史实。
- 如关键信息会影响全文，向用户索要来源；否则先交付并在编辑备注中标明待核验项。

不要用“据传”“有一种说法”给缺乏来源的故事披上事实外衣。资料少时，诚实比故事完整更重要。

### 3. 确定独立角度

先用一句话概括文章角度，例如：

> 从校园广播里的《童年》，写一代人如何第一次意识到时间会过去。

角度应同时包含“这首歌为什么值得写”和“它与读者有什么关系”。避免只按“发行信息 -> 创作背景 -> 歌词 -> 感想”机械拼装。

### 4. 组织正文

单曲文章默认 900-1400 个中文字符，用户另有要求时从其要求。根据素材选择 4-6 个内容模块，不强制每篇都长成同一个五段式：

1. 场景钩子：用一个与歌曲年代匹配的具体瞬间进入
2. 歌曲坐标：交代版本、年份、专辑或影视关系
3. 核心故事：只写核实过且真正服务主题的背景
4. 听感或文本分析：解释旋律、演唱、歌词为何有效
5. 代际共鸣：连接普通人的成长、离别、友情或家庭经验
6. 余味与互动：用具体问题邀请读者补充记忆

素材不足时宁可缩短文章，也不要用空泛抒情填满篇幅。

### 5. 做发布前检查

依次检查事实、版权、用户约束、结构和文风。发现未核验信息时，删掉、弱化为明确的主观分析，或放进“编辑备注”，不能把风险藏在流畅文案里。

## 版权与平台边界

法律和平台判断取决于使用目的、篇幅、来源标注、授权情况和具体呈现方式，不存在仅按句数计算的自动安全线。需要解释规则或处理高风险素材时，阅读 [references/compliance.md](references/compliance.md)。

执行以下保守默认值：

- 只有在评论确实需要时才引用歌词，优先引用 1 个短片段。
- 单篇默认最多使用 3 个短歌词片段；这是编辑上限，不是法律免责标准。
- 引用应服务于逐句评论，不能拼接成歌曲主要内容，也不能用改写、翻译或图片绕过限制。
- 在正文适当位置注明歌曲名及词作者；不确定词作者时不引用歌词。
- 不提供未授权音频、MV、网盘、下载地址、破解方式或“评论区取资源”等引导。
- 不自动插入“点击收听”“扫码听完整版”。用户明确拥有授权或使用平台官方音乐能力时，才可建议在后台配置，并提醒其确认授权与平台规则。
- 不把专辑封面、歌手照片、影视截图视为可自由使用的素材；若涉及配图，只给出原创视觉方向或提醒确认授权。
- 不大段复述其他文章、传记或访谈。引用他人观点时标明来源，文章主体必须是新的组织、分析和表达。

如果用户要求大量歌词或资源链接，简短解释边界，然后改用歌词主题概述、听感分析和记忆叙事完成任务。

## 事实与表述规则

写作前在内部把信息分成三类：

- 已核实事实：可直接陈述
- 用户提供信息：可使用，但有明显疑点时提醒核验
- 作者分析或共同记忆：用“听起来”“许多人会想起”“对不少读者而言”等措辞表明性质

不得编造：

- 创作动机、录音现场、人物对话
- 销量、榜单、奖项、播放量
- “人人都会唱”“影响了一代人”等无法证明的绝对判断
- 歌手私生活、负面传闻和与文章无关的争议

对歌曲发行时间存在多个口径时，准确说明口径，例如“最早发表于……，后收录于……”，不要擅自选一个年份。

## 文风

### 要做到

- 温和、克制、有生活温度
- 让抽象情绪落在可感知的物件、声音、动作和场景里
- 年代符号与歌曲传播时期匹配
- 句式有长短变化，每段通常 2-4 句
- 读者不熟悉背景也能看懂
- 保留用户已有文风中真实、有辨识度的部分

可选时代线索只用于启发，不要每篇照抄：

- 70/80 年代：收音机、磁带、音像店、黑白或彩色电视、绿皮火车
- 90 年代：随身听、校园广播、点歌台、录像厅、港台剧
- 00 年代：MP3、CD 店、QQ 空间、贴吧、选秀节目、彩铃
- 10 年代早期：智能手机初代、微博、音乐播放器歌单、校园社团

先判断歌曲在目标受众中的实际传播时期，再选符号。发行于 1980 年代的歌也可能因 1990 年代翻唱走红，不能仅按首发年份套场景。

### 要避免

- 连续使用“青春、岁月、回忆、时光”而没有新信息
- “听懂已是曲中人”“前奏一响泪目”等批量化标题和句式
- 虚构“那年夏天我们都……”式集体记忆
- 过度感叹号、夸张排比和强行催泪
- 每段都总结中心思想
- 把互动写成“点赞在看转发三连”式口播

## 标题策略

标题先突出歌曲或专题，再呈现具体冲突、记忆或发现。默认给 1 个主标题；用户要求备选时给 5 个，且至少覆盖以下不同角度中的 3 种：

- 时代场景
- 创作或版本故事
- 歌曲里的具体情绪
- 影视、校园或家庭记忆
- 反常识信息

标题应准确，不制造虚假稀缺、夸大传播度或杜撰幕后。少用竖线和固定模板，避免连续文章标题高度相似。

## 各任务输出

### 单曲文章

```text
【标题】
[主标题]

【正文】
[可直接发布的正文]

【编辑备注】（仅在有待核验项或用户要求来源时出现，不属于正文）
[待核验信息或参考来源]
```

正文应自足。不要在开头解释写作方法，也不要把自检清单输出给读者。

### 优化改写

先完整通读原文，优先处理事实错误、过量引用、未经授权资源引导和不匹配的年代符号，再优化文笔。

```text
【优化后推文】
[完整成稿]

【主要改动】
- [最多 3-5 条真正重要的改动]

【待确认】（没有则省略）
- [无法核实但会影响发布的信息]
```

保留用户的核心观点、第一手经历和有辨识度的句子。除非用户要求，不要为了套模板重写成另一篇文章。

### 标题方案

```text
【标题方案】
1. ...
2. ...
3. ...
4. ...
5. ...

【推荐】
[推荐 1 个，并用一句话说明它与文章角度的匹配]
```

### 系列选题

根据频率计算目标篇数。例如“每周 3 篇、一个月”默认目标为 12 篇；若用户指定自然月，则按实际周次说明口径。严格筛选下，只有核验通过的项目才计入已交付篇数，准确性高于凑足目标数量。

每个正选项至少包含：歌曲、歌手或版本、首次发行年份、主题角度、选择理由。严格筛选任务还要附与编号对应的核验来源。按周或主题组织，避免整份计划只有爱情歌曲。

最后补充：

- 发布节奏
- 主题配比
- 需要进一步核验的版本或年份

### 局部润色

只改用户指定部分，不擅自扩写整篇。输出“润色后”与 1-3 条改动说明；除非用户要求对照，不重复展示原文。

### 合集文章

每首歌只保留一个清晰观点，避免写成多篇单曲文章的简单拼接。合集默认不逐首引用歌词；如确有必要，全篇仍执行最少必要引用原则。

## 质量门槛

提交前逐项检查：

- [ ] 歌名、歌手、版本、年份、专辑和词曲作者没有混淆
- [ ] 所有幕后故事、数据和引语都有可靠依据，或已删除/标注待核验
- [ ] 歌词引用是评论所必需的，足够短，且注明歌曲与词作者
- [ ] 没有未授权资源、下载引导或权利不明的配图建议
- [ ] 用户指定的年代、语种、歌手、主题和数量全部满足
- [ ] 正选歌单逐首核对，越界项只在备选中出现
- [ ] 文章有一个明确角度，不是资料和金句的流水账
- [ ] 年代细节与歌曲实际传播时期匹配
- [ ] 段落适合手机阅读，没有连续大段文字
- [ ] 标题准确、有区分度，不依赖陈词滥调
- [ ] 结尾问题具体，读者可以用自己的经历回答

最终标准：用户拿到的是可信、克制、有辨识度的成稿；任何不能确认的事实都不会因为文笔流畅而被悄悄写成真的。

## 配图功能

### 概述

为怀旧音乐公众号文章添加符合主题的插图，增强视觉吸引力和情感共鸣。配图应与文章年代感、情绪氛围和内容主题保持一致。

### 触发时机

- 用户明确要求"配图""加图片""插图""生成图片"
- 完成文章创作后，主动询问："需要为这篇文章配图吗？"
- 用户提供已有文章并要求视觉化

### 配图流程

**使用 TaskList 管理**：配图流程包含 8 个步骤，必须使用 TaskList 让用户看到进度。

#### 启动流程

```
开始配图流程前，创建任务清单：
1. 分析文章结构
2. 推荐配图方案
3. 确定视觉风格
4. 生成配图大纲
5. 生成提示词文件
6. 生成图片
7. 插入图片到文章
8. 输出完成总结

预计耗时：3-5 分钟
```

#### 1. 分析文章结构

**TaskCreate**: subject="分析文章结构", activeForm="分析文章结构中"

读取文章内容，识别：
- 文章主题（单曲故事/歌手专题/歌单/时代回忆）
- 年代背景（70/80/90/00年代）
- 情感基调（温暖/怀旧/青春/伤感）
- 章节结构和长度
- 关键场景和意象

#### 2. 推荐配图方案

### 版权注意事项

配图功能严格遵守版权原则：

**不允许生成**：
- 真实歌手的肖像或面容
- 专辑封面的复制或模仿
- 明显可识别的 MV 截图或场景重现
- 受版权保护的 logo、字体、品牌标识

**推荐生成**：
- 抽象的年代场景（教室、街道、音像店）
- 象征性物件（收音机、磁带、随身听）
- 情感氛围图（黄昏、校园、火车站）
- 时间线和信息图（数据、年份、关系图）
- 无具体人脸的剪影或背影

生成提示词时，使用"一个人的背影""剪影""抽象化的人物"而非"歌手肖像""专辑封面"。

### 常见问题

#### Q: 如何获取微信公众号 AppID 和 AppSecret？

**A**: 
1. 登录微信公众平台 (mp.weixin.qq.com)
2. 进入"设置与开发" -> "基本配置"
3. 在"开发者 ID(AppID)"处查看 AppID
4. 在"开发者密码(AppSecret)"处点击"重置"或查看
5. 注意：AppSecret 仅显示一次，请妥善保管

#### Q: 可以直接发布而不创建草稿吗？

**A**: 
默认创建草稿是安全策略。草稿允许：
- 在后台预览效果
- 进行最后调整
- 选择发布时间
- 避免误操作

如需直接发布，需要在微信公众号后台手动操作。

#### Q: 图片上传后在哪里？

**A**: 
图片作为"永久素材"上传到公众号素材库：
1. 登录后台
2. 进入"内容与互动" -> "素材管理"
3. 在"图片"标签下查看

#### Q: 封面图尺寸要求是什么？

**A**: 
- 推荐比例：2.35:1（宽封面）
- 推荐尺寸：900x383 像素
- 格式：JPG、PNG
- 大小：< 1MB

配图功能生成的图片默认使用 16:9，可以裁剪为 2.35:1。

#### Q: 如何测试不实际发布？

**A**: 
使用 `--dry-run` 选项：
```bash
node scripts/publish-to-wechat.js --article article.md --dry-run --output test.html
```
这会生成 HTML 文件但不上传到微信。

#### Q: 发布失败如何排查？

**A**: 
按以下顺序检查：
1. 检查网络连接
2. 验证 AppID 和 AppSecret
3. 确认公众号类型（需要认证服务号或订阅号）
4. 检查图片路径和格式
5. 查看完整错误信息

#### Q: 可以批量发布多篇文章吗？

**A**: 
可以编写脚本循环调用：
```bash
for article in *.md; do
  node scripts/publish-to-wechat.js --article "$article" --auto-cover
done
```

2. **编辑草稿**

   - 修改标题、摘要、封面
   - 添加音频、视频等多媒体

3. **发布草稿**
   - 点击"发表"
   - 选择发布时间（立即/定时）
   - 选择推送对象
   - 确认发布

### 安全注意事项

1. **凭证保护**
   - 不要将 AppID 和 AppSecret 提交到代码仓库
   - 使用 `.env` 文件并加入 `.gitignore`
   - 定期更新 AppSecret

2. **内容审核**
   - 发布前人工复查内容
   - 确保符合微信公众平台运营规范
   - 注意敏感词和违规内容

3. **图片版权**
   - 确保所有图片有使用权
   - 配图功能生成的图片已考虑版权安全
   - 不要使用侵权的封面图

4. **草稿管理**
   - 定期清理未使用的草稿
   - 素材库有数量限制（图片 100,000 张）
   - 及时删除不需要的素材
