# 统一响应格式

> 适用范围：所有 HTTP API 接口响应，适用于产品经理定义接口规范、研发人员实现接口

---

## 1. 规则说明

统一的响应格式能够简化前后端对接，提高接口的可维护性和可读性。所有接口必须使用统一的响应封装格式 `Result<T>`，确保响应结构的一致性。

## 2. 规则内容

### 2.1 标准响应结构

所有接口必须返回以下结构：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": { ... },
  "traceId": "a1b2c3d4e5f6"
}
```

### 2.2 字段说明

| 字段名 | 类型 | 必返 | 说明 |
|-------|------|------|------|
| code | int | 是 | 业务状态码，200表示成功，其他表示失败 |
| msg | string | 是 | 响应信息，成功时为"操作成功"，失败时为具体错误原因 |
| data | T | 是 | 响应数据，成功时返回具体数据，失败时为 null |
| traceId | string | 是 | 链路追踪ID，用于日志排查和问题定位 |

### 2.3 响应码规范

| 状态码范围 | 说明 | 使用场景 |
|----------|------|---------|
| 200 | 成功 | 操作成功 |
| 400-499 | 客户端错误 | 参数错误、业务规则校验失败 |
| 500-599 | 服务端错误 | 系统异常、第三方服务异常 |

## 3. 示例

### 正确示例

**成功响应：**
```json
{
  "code": 200,
  "msg": "查询成功",
  "data": {
    "userId": 10001,
    "nickname": "张三",
    "email": "zhangsan@example.com"
  },
  "traceId": "a1b2c3d4e5f6"
}
```

**业务异常响应：**
```json
{
  "code": 400,
  "msg": "用户名或密码错误",
  "data": null,
  "traceId": "a1b2c3d4e5f6"
}
```

**系统异常响应：**
```json
{
  "code": 500,
  "msg": "系统繁忙，请稍后重试",
  "data": null,
  "traceId": "a1b2c3d4e5f6"
}
```

**分页查询响应：**
```json
{
  "code": 200,
  "msg": "查询成功",
  "data": {
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "pages": 10,
    "list": [
      { "userId": 1, "nickname": "用户1" },
      { "userId": 2, "nickname": "用户2" }
    ]
  },
  "traceId": "a1b2c3d4e5f6"
}
```

### 错误示例

```json
// ❌ 缺少 traceId 字段
{
  "code": 200,
  "msg": "操作成功",
  "data": { ... }
}

// ❌ 直接返回数据，没有统一封装
{
  "userId": 10001,
  "nickname": "张三"
}

// ❌ 使用非标准字段名
{
  "status": 200,
  "message": "操作成功",
  "result": { ... }
}
```

## 4. 检查清单

- [ ] 所有接口返回统一的 Result<T> 结构
- [ ] 包含 code、msg、data、traceId 四个字段
- [ ] code 字段使用标准业务状态码
- [ ] msg 字段提供清晰的提示信息
- [ ] 失败时 data 字段为 null
- [ ] traceId 字段用于链路追踪
- [ ] 系统异常不暴露堆栈信息给前端
