# 验收标准格式

> 适用范围：所有功能需求的验收标准定义，适用于产品经理、测试工程师

---

## 1. 规则说明

验收标准（Acceptance Criteria，AC）是判断功能是否符合预期的标准。清晰的验收标准能够帮助开发人员理解需求，帮助测试人员设计测试用例。本规范定义了验收标准的标准格式和最佳实践。

## 2. 规则内容

### 2.1 Given-When-Then 格式

使用 **Given-When-Then（Gherkin）** 格式定义验收标准：

```
Given（前置条件）：描述系统的初始状态
When（触发动作）：用户或系统执行的操作
Then（预期结果）：系统应有的反应（含界面、数据库、接口）
```

### 2.2 格式说明

| 关键字 | 说明 | 示例 |
|-------|------|------|
| Given | 前置条件，描述系统的初始状态 | Given 用户已登录，且有余额 100 元 |
| When | 触发动作，用户或系统执行的操作 | When 用户点击"立即支付"按钮 |
| Then | 预期结果，系统应有的反应 | Then 用户余额扣减为 20 元，生成订单 |
| And | 补充条件或结果 | And 页面跳转至支付成功页 |

### 2.3 验收标准原则

1. **可测试**：验收标准必须是可测试的，能够通过测试用例验证
2. **明确具体**：避免模糊的描述，使用具体的数值和状态
3. **完整覆盖**：覆盖正常流程、异常流程、边界值
4. **独立性**：每个验收标准应该是独立的，不依赖其他验收标准

### 2.4 验收标准与测试用例的关系

- 一个验收标准对应一个或多个测试用例
- 测试用例是验收标准的具体实现
- 所有验收标准通过，功能才算验收通过

## 3. 示例

### 正确示例

**示例 1：用户登录**

**AC1：正常登录**
```
Given 用户在登录页面
  And 用户名为"zhangsan"，密码为"123456"
  And 该用户账号状态为"正常"
When 用户输入用户名和密码
  And 点击"登录"按钮
Then 登录成功，页面跳转至首页
  And 显示用户昵称"张三"
  And 数据库记录登录日志
  And 生成登录 Token，有效期 2 小时
```

**AC2：用户名或密码错误**
```
Given 用户在登录页面
  And 用户名为"zhangsan"，密码为"wrong_password"
When 用户输入用户名和密码
  And 点击"登录"按钮
Then 登录失败，页面提示"用户名或密码错误"
  And 页面不跳转，停留在登录页面
  And 数据库不记录登录日志
  And 不生成登录 Token
```

**AC3：账号被禁用**
```
Given 用户在登录页面
  And 用户名为"zhangsan"，密码为"123456"
  And 该用户账号状态为"禁用"
When 用户输入用户名和密码
  And 点击"登录"按钮
Then 登录失败，页面提示"账号已被禁用，请联系管理员"
  And 页面不跳转，停留在登录页面
  And 数据库记录登录尝试日志
  And 不生成登录 Token
```

**示例 2：创建订单**

**AC1：正常创建订单**
```
Given 用户已登录，用户 ID 为 10001
  And 用户余额为 100 元
  And 购物车商品总价为 80 元
  And 商品库存充足
When 用户点击"立即支付"按钮
Then 用户余额扣减为 20 元
  And 生成订单，订单状态为"已支付"
  And 商品库存对应减少
  And 页面跳转至支付成功页
  And 数据库订单表新增一条记录
  And 发送订单创建成功的短信通知
```

**AC2：余额不足**
```
Given 用户已登录，用户 ID 为 10001
  And 用户余额为 50 元
  And 购物车商品总价为 80 元
When 用户点击"立即支付"按钮
Then 支付失败，页面提示"余额不足，请充值"
  And 用户余额不变，仍为 50 元
  And 不生成订单
  And 商品库存不变
  And 页面不跳转，停留在当前页面
```

**AC3：库存不足**
```
Given 用户已登录，用户 ID 为 10001
  And 用户余额为 100 元
  And 购物车商品总价为 80 元
  And 商品库存为 0
When 用户点击"立即支付"按钮
Then 支付失败，页面提示"商品库存不足"
  And 用户余额不变，仍为 100 元
  And 不生成订单
  And 页面不跳转，停留在当前页面
```

**示例 3：分页查询**

**AC1：正常分页查询**
```
Given 数据库中有 100 条用户记录
  And 用户在用户列表页面
When 用户访问第 1 页，每页 10 条
Then 页面显示 10 条用户记录
  And 显示总记录数为 100
  And 显示总页数为 10
  And 显示当前页码为 1
  And 分页组件显示"上一页"按钮为禁用状态
  And 分页组件显示"下一页"按钮为可用状态
```

**AC2：查询最后一页**
```
Given 数据库中有 100 条用户记录
  And 用户在用户列表页面
When 用户访问第 10 页，每页 10 条
Then 页面显示 10 条用户记录
  And 显示总记录数为 100
  And 显示总页数为 10
  And 显示当前页码为 10
  And 分页组件显示"上一页"按钮为可用状态
  And 分页组件显示"下一页"按钮为禁用状态
```

**AC3：查询空结果**
```
Given 数据库中有 0 条用户记录
  And 用户在用户列表页面
When 用户访问第 1 页，每页 10 条
Then 页面显示"暂无数据"
  And 显示总记录数为 0
  And 显示总页数为 0
  And 不显示分页组件
```

### 错误示例

```
# ❌ 描述不明确
Given 用户已登录
When 用户点击按钮
Then 操作成功

# ❌ 缺少前置条件
When 用户点击"立即支付"按钮
Then 支付成功

# ❌ 缺少具体数值
Given 用户有余额
When 用户购买商品
Then 余额减少

# ❌ 缺少数据库变化
Given 用户已登录
When 用户点击"立即支付"按钮
Then 页面跳转至支付成功页  # 缺少数据库订单记录

# ❌ 不可测试
Given 用户已登录
When 用户点击"立即支付"按钮
Then 用户体验良好  # 无法测试
```

## 4. 检查清单

- [ ] 使用 Given-When-Then 格式
- [ ] 前置条件（Given）是否明确？
- [ ] 触发动作（When）是否清晰？
- [ ] 预期结果（Then）是否完整？
- [ ] 是否包含界面反馈？
- [ ] 是否包含数据库变化？
- [ ] 是否包含接口返回？
- [ ] 是否覆盖正常流程？
- [ ] 是否覆盖异常流程？
- [ ] 是否覆盖边界值？
- [ ] 验收标准是否可测试？
- [ ] 验收标准是否独立？
