# 测试覆盖矩阵

> 适用范围：所有功能测试用例设计，适用于测试工程师

---

## 1. 规则说明

测试覆盖矩阵定义了测试用例必须覆盖的维度，确保测试的全面性和有效性。本规范定义了测试覆盖的各个维度、优先级分配和覆盖率要求。

## 2. 规则内容

### 2.1 测试维度

| 测试维度 | 说明 | 对应优先级 | 覆盖率要求 |
|---------|------|---------|----------|
| **正常流程（Happy Path）** | 主成功场景，所有条件满足时的正常操作 | P0 | 100% |
| **业务异常** | 库存不足、余额不足、权限不足等业务规则拦截 | P0/P1 | 100% |
| **参数边界值** | 长度上限/下限、数值范围、空值 | P1/P2 | 80% |
| **安全性** | SQL注入、XSS注入、越权访问 | P0/P1 | 100% |
| **幂等性** | 重复提交、网络重试 | P1 | 100% |
| **并发场景** | 多用户同时操作同一资源 | P1 | 80% |
| **兼容性** | 主流浏览器/设备（iOS/Android/PC） | P2 | 80% |
| **性能/体验** | 弱网、大数据量、加载时间 | P2 | 60% |

### 2.2 正常流程测试

**测试内容：**
- 主成功场景，所有条件满足时的正常操作
- 用户按照预期路径完成操作

**测试用例示例：**
```
TC_LOGIN_001：正常登录-有效账号密码
前置条件：用户在登录页面，账号状态正常
测试步骤：
1. 输入有效的用户名和密码
2. 点击"登录"按钮
预期结果：
1. 登录成功，页面跳转至首页
2. 显示用户昵称
3. 数据库记录登录日志
```

### 2.3 业务异常测试

**测试内容：**
- 业务规则校验失败的场景
- 库存不足、余额不足、权限不足等

**测试用例示例：**
```
TC_ORDER_002：创建订单-余额不足
前置条件：用户已登录，余额 50 元，商品总价 80 元
测试步骤：
1. 点击"立即支付"按钮
预期结果：
1. 支付失败，提示"余额不足，请充值"
2. 用户余额不变
3. 不生成订单
```

### 2.4 参数边界值测试

**测试内容：**
- 参数的最小值、最大值、最小值-1、最大值+1
- 空值、null、空字符串

**测试用例示例：**
```
TC_USER_003：用户注册-用户名长度边界值
前置条件：用户在注册页面
测试数据：
- 最小值：6 位（"abc123"）
- 最大值：16 位（"abcdefghij123456"）
- 最小值-1：5 位（"abc12"）
- 最大值+1：17 位（"abcdefghij1234567"）
预期结果：
- 6-16 位：注册成功
- 5 位：提示"用户名长度至少 6 位"
- 17 位：提示"用户名长度不能超过 16 位"
```

### 2.5 安全性测试

**测试内容：**
- SQL 注入、XSS 注入
- 越权访问、CSRF 攻击
- 敏感信息泄露

**测试用例示例：**
```
TC_SECURITY_001：SQL 注入防护
前置条件：用户在登录页面
测试步骤：
1. 在用户名输入框输入：' OR '1'='1
2. 在密码输入框输入：' OR '1'='1
3. 点击"登录"按钮
预期结果：
1. 登录失败，提示"用户名或密码错误"
2. 系统拦截并记录安全日志
3. 不暴露堆栈信息
```

### 2.6 幂等性测试

**测试内容：**
- 重复提交、网络重试
- 消息队列重复消费

**测试用例示例：**
```
TC_IDEMPOTENCY_001：创建订单-重复提交
前置条件：用户已登录，余额充足，库存充足
测试步骤：
1. 快速点击"立即支付"按钮 2 次
预期结果：
1. 仅生成 1 条订单记录
2. 第二次请求被拦截，提示"请勿重复提交"
3. 用户余额只扣减一次
```

### 2.7 并发场景测试

**测试内容：**
- 多用户同时操作同一资源
- 库存扣减、余额扣减等

**测试用例示例：**
```
TC_CONCURRENCY_001：库存扣减-并发场景
前置条件：商品库存为 10
测试步骤：
1. 10 个用户同时购买该商品，每人购买 1 件
预期结果：
1. 10 个用户都购买成功
2. 商品库存变为 0
3. 不出现超卖现象
```

### 2.8 兼容性测试

**测试内容：**
- 主流浏览器（Chrome、Safari、Firefox）
- 主流设备（iOS、Android、PC）
- 不同分辨率

**测试用例示例：**
```
TC_COMPATIBILITY_001：用户登录-浏览器兼容性
前置条件：用户在登录页面
测试环境：
- Chrome 最新版
- Safari 最新版
- Firefox 最新版
测试步骤：
1. 在不同浏览器中输入用户名和密码
2. 点击"登录"按钮
预期结果：
1. 所有浏览器都能正常登录
2. 页面显示正常，无样式错乱
```

### 2.9 性能/体验测试

**测试内容：**
- 弱网环境、大数据量
- 页面加载时间、接口响应时间

**测试用例示例：**
```
TC_PERFORMANCE_001：订单列表查询-大数据量
前置条件：数据库中有 10000 条订单记录
测试步骤：
1. 访问订单列表页面
2. 查询第 1 页，每页 10 条
预期结果：
1. 页面加载时间 < 2 秒
2. 接口响应时间 < 500ms
3. 页面显示正常，无卡顿
```
