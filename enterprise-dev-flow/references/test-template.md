## Part 1：测试用例模板

### 用例结构定义

每个测试用例必须包含以下字段：

| 字段 | 说明 |
|------|------|
| **用例编号（ID）** | 唯一标识，格式：`TC_[模块]_[序号]`，如 `TC_LOGIN_001` |
| **用例标题（Title）** | 简明描述测试点，如"正常登录-有效账号密码" |
| **优先级（Priority）** | P0/P1/P2/P3（参考 `rules/testing/defect-severity.md`） |
| **前置条件（Pre-condition）** | 执行前必须满足的环境/数据状态 |
| **测试步骤（Steps）** | 清晰的操作指令，编号列出 |
| **测试数据（Data）** | 具体的输入值（使用测试账号，禁止生产数据）|
| **预期结果（Expected Result）** | 系统应有的反应（含界面提示、数据库变化、接口返回） |
| **实际结果（Actual Result）** | 执行时填写 |
| **状态（Status）** | Pass / Fail / Block |

---

### 测试用例设计原则

设计时必须综合使用以下方法：

1. **等价类划分**：将输入分为有效等价类和无效等价类，从每类中取代表值
2. **边界值分析**：重点测试最小值、最大值、最小值-1、最大值+1
3. **场景法**：从用户故事出发，覆盖主成功场景、扩展场景、异常场景
4. **错误推测法**：基于经验推测容易出错的地方（并发、空值、特殊字符）

---

### 测试用例覆盖矩阵（必须覆盖的维度）

**核心覆盖维度：**
- 正常流程（Happy Path）：主成功场景，所有条件满足时的正常操作（P0）
- 业务异常：库存不足、余额不足、权限不足等业务规则拦截（P0/P1）
- 参数边界值：长度上限/下限、数值范围、空值（P1/P2）
- 安全性：SQL注入、XSS注入、越权访问（P0/P1）
- 幂等性：重复提交、网络重试（P1）
- 并发场景：多用户同时操作同一资源（P1）
- 兼容性：主流浏览器/设备（iOS/Android/PC）（P2）
- 性能/体验：弱网、大数据量、加载时间（P2）

**详细规范参考：** `rules/testing/test-coverage-matrix.md`
