# 企业级产品开发通用规范

> 本文件是从PRD规范、研发详细设计规范、测试规范中抽取的**跨角色共用规则**。
> 产品经理、研发人员、测试工程师在各自工作时均需遵循。

---

## 1. 安全红线（Security Red Lines）

**任何角色书写的文档和代码均不得违反以下规则：**

核心要点：
- 敏感信息保护（密码、手机号、身份证等必须脱敏）
- SQL 安全（禁止 SQL 注入，禁止 `SELECT *`）
- 越权访问防护（必须校验权限）
- 错误信息安全（不暴露堆栈信息）

**详细规范参考：** `rules/security/security-red-lines.md`
**敏感数据处理：** `rules/security/sensitive-data.md`

---

## 2. 幂等性设计（Idempotency）

**适用于所有写操作场景（产品设计、接口开发、测试用例）：**

核心要点：
- 同一操作重复发起多次，产生的结果与执行一次相同
- 必须设计幂等的场景：用户连续点击、网络重试、消息队列重复消费
- 常见实现方式：数据库唯一索引、Redis 原子锁、前端令牌机制、状态机版本号控制

**详细规范参考：** `rules/code-quality/idempotency.md`

---

## 3. 状态机规范（State Machine）

**适用于包含状态流转的业务实体（订单、审批、任务等）：**

核心要点：
- 文档中必须明确四要素：状态列表、触发事件、动作、约束条件
- 状态流转必须有明确的前提条件和执行操作
- 防止非法状态流转

**详细规范参考：** `rules/code-quality/state-machine.md`

---

## 4. 缺陷等级标准（Defect Severity）

**测试发现的缺陷统一按以下标准分级，产品和研发在设计时也需预防对应等级的问题：**

核心要点：
- P0（致命）：系统崩溃、数据丢失、核心流程阻断，2小时内修复
- P1（严重）：主要功能错误，无临时解决方案，24小时内修复
- P2（一般）：次要功能错误，有临时方案，版本发布前修复
- P3（轻微）：建议性问题、文案错误，视资源情况排期

**详细规范参考：** `rules/testing/defect-severity.md`

---

## 5. 验收标准格式（Acceptance Criteria）

**产品经理在PRD中定义的AC，应与测试工程师的测试用例直接对应：**

核心要点：
- 使用 Given-When-Then（Gherkin）格式
- Given（前置条件）：描述系统的初始状态
- When（触发动作）：用户或系统执行的操作
- Then（预期结果）：系统应有的反应（含界面、数据库、接口）

**详细规范参考：** `rules/testing/acceptance-criteria.md`

---

## 6. 非功能性需求基线（NFR Baseline）

**凡产品需求涉及以下场景，必须明确指标，不得缺失：**

核心要点：
- 性能：P99响应时间、QPS峰值、并发用户数（核心接口P99 < 200ms）
- 可用性：SLA目标（99.99%）
- 安全性：认证方式、加密策略、频率限制
- 容灾：降级方案、熔断阈值、回滚计划
- 可观测性：日志TraceID、监控告警阈值、埋点

**详细规范参考：** `rules/architecture/nfr-baseline.md`

---

## 7. 数据字典规范（Data Dictionary）

**产品经理定义业务字段，研发设计数据库字段，测试设计边界值，三者需保持一致：**

标准字段表格格式：

| 字段标识 | 字段名称 | 类型/长度 | 必填 | 默认值 | 校验规则 | 数据来源 | 敏感级 |
|---------|---------|---------|------|-------|---------|---------|------|
| `user_id` | 用户ID | BigInt | 是 | - | >0 | 登录态 | L3（高） |
| `amount` | 交易金额 | Decimal(18,2) | 是 | 0.00 | >=0.01 | 用户输入 | L2（中） |
| `status` | 状态 | TinyInt | 是 | 0 | Enum(0,1,2) | 系统生成 | - |

**敏感级别说明：**
- L3（高）：身份证号、银行卡号、密码等，存储加密+展示脱敏
- L2（中）：手机号、金额等，展示脱敏
- L1（低）：普通业务数据，无特殊要求

## 8. 发布安全规范（Release Safety）

**灰度发布顺序：**
```
用户白名单（内测） → 5% → 20% → 50% → 100%
```

**每次变更必须有明确的回滚方案：**
- 数据库变更：回滚SQL脚本
- 接口变更：新旧API共存，不强制升级
- 配置变更：Nacos历史版本可一键回滚
- 上线失败的回滚时间目标（RTO）：≤ 30分钟

## 9. 代码提交规范（Git Commit）

**提交信息格式：** `<type>: <subject>`

核心要点：
- 提交类型：feat（新功能）、fix（修复）、docs（文档）、refactor（重构）等
- 提交主题：使用中文描述，简明扼要，不超过 50 个字符
- 分支命名：feature/、bugfix/、hotfix/、release/ 等前缀
- 提交前检查：类型正确、代码已测试、符合编码规范

**详细规范参考：**
- 提交信息规范：`rules/git/commit-message.md`
- 分支命名规范：`rules/git/branch-naming.md`
- 提交前检查清单：`rules/git/commit-checklist.md`

## 10. 异常处理规范（Exception Handling）

**适用于产品经理定义错误场景、研发人员实现异常处理、测试工程师编写异常测试用例：**

核心要点：
- 业务异常统一使用 `BizException(code, message)`
- 异常编码规则：模块编码（两位，从10开始）+ 业务错误码（四位，从0001开始）
- 使用 i18n 管理错误信息，支持多语言
- 统一异常处理器：`@RestControllerAdvice` 全局捕获异常
- 特殊错误码：400000（参数校验）、401000（未认证）、403000（无权限）、500000（系统错误）

**详细规范参考：** `rules/code-quality/exception-handling.md`

## 11. 接口设计规范摘要（API Design）

**适用于产品经理定义接口需求、研发人员设计接口、测试工程师编写接口测试用例：**

核心要点：
- 统一响应格式：`Result<T>`（包含 code、msg、data、traceId）
- URL 命名：`/{api-prefix}/{version}/{resource}/{action}`，资源名称用复数
- 响应码规范：HTTP 状态码 + 业务状态码（如 `ORDER_001`）
- 参数校验：所有请求参数必须定义校验规则（长度、范围、格式、必填）
- 分页查询：标准分页参数（pageNum、pageSize），禁止深分页
- 接口安全：认证授权、敏感信息脱敏、防重放攻击

**详细规范参考：**

- URL 命名规范：`rules/api/url-naming.md`
- RESTful 设计：`rules/api/restful-design.md`
- 响应码规范：`rules/api/error-codes.md`

- 分页查询规范：`rules/api/pagination.md`
