# 上下文工程：为什么 Prompt 不够用了

过去两年，大家习惯把和大模型协作的能力等同于"会写 Prompt"。但当模型进入真实的工程系统，决定效果上限的往往不是某一句提示词，而是模型能稳定读到哪些上下文、以什么结构读到。

> 上下文工程的核心不是"写得更花哨"，而是"让模型每次都能拿到正确、最新、可验证的信息"。

## 一、从 Prompt 到 Context

单轮 Prompt 假设所有信息都塞进一段文字里。一旦任务涉及多个文件、历史决策、外部数据，这种假设就崩了。上下文工程把关注点从"这句话怎么说"转移到"这套信息怎么组织"。

核心判断：如果换一个人接手同样的上下文也能复现结果，说明上下文是工程资产；如果只有作者本人知道怎么喂提示词，那它还停留在手艺阶段。

### 检索不是银弹

很多团队上来就接一个向量库做 RAG，然后发现召回的片段经常答非所问。问题通常不在模型，而在于切分粒度、元数据和重排策略。

```
原始文档
  ↓ 结构化切分（按标题/语义，而非定长）
  ↓ 附加元数据（来源、时间、权限）
  ↓ 检索 + 重排
  ↓ 拼装进上下文窗口
```

## 二、落地时容易忽略的三件事

把上下文工程做扎实，比追逐新模型更能拉开差距：

- 时效性：过期文档比没有文档更危险，要带时间戳并定期淘汰
- 权限边界：哪些上下文能进模型，必须有明确的访问控制
- 可观测性：记录每次调用实际喂了什么，出问题才能复盘
