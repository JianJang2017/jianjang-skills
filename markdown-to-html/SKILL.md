# Markdown 转 HTML 排版

把一篇 Markdown 文档转换成**排版精美、可直接粘贴到微信公众号编辑器或邮件客户端**的 HTML。核心约束是：所有样式必须内联在 `style="..."` 属性里，不能依赖 `<style>` 块、外部样式表或 `class` 选择器——因为公众号编辑器和多数邮件客户端会把这些统统丢掉，只保留内联样式。

技能内置多套主题，每套主题是一份"组件库"：把 Markdown 的各种元素（标题、段落、引用、列表、代码、表格、图片）以及一些语义化组件映射到一段固定的内联 HTML 片段。你的工作是读懂 Markdown，选好主题，逐元素套用组件，拼成完整 HTML。

## 工作流程

1. **拿到 Markdown**：用户给文件路径就读文件；给的是一段文本就直接用。
2. **看有哪些主题**：`references/themes/` 下每个 `theme-<名字>.md` 就是一套主题，开头一句话写明它的适用场景或视觉定位。先扫一遍这些首行，了解可选项——不要假设只有某几套。
3. **选主题**：用户点名了主题就用它；没点名就按下面的"主题选择"判断，并**在回复里说明选了哪套、为什么**，让用户能一句话换。
4. **读主题文件**：打开选中的 `theme-<名字>.md`，里面有页面外壳和每个组件的精确 HTML 片段；组件名称由该主题自己定义，以文件为准。同名 `references/themes/examples/<名字>.html` 是完整范例，拿不准整体效果时对照它。
5. **逐元素映射**：按文档顺序，把每个 Markdown 元素替换成该主题里对应组件的 HTML，填入内容。
6. **拼装**：把所有组件片段塞进主题"页面外壳"的 `{{content}}` 位置，`{{title}}` 用文章标题。
7. **校验并交付**：写出 `.html` 文件，跑一遍 `scripts/validate_html.py` 确认没有会掉样式的写法，告诉用户文件路径和所选主题。

## 主题选择

`references/themes/` 内置 40 套主题，分两类，选法不同：

- **题材型主题**：首行写明"适用于……"，为特定文体设计，错配会很违和。
  - **原生蓝图（native-blueprint）**：技术长文、AI/工程范式、架构方法论、技术战略、深度分析。
  - **青格笔记（cyan-grid）**：备考计划、学习复盘、资料清单、知识整理、教程笔记。
- **风格型主题**：首行写"视觉定位：……，不绑定任何文章题材"或"视觉定位：……。主题由 markdown2wechat 的 XXX 转换而来"，是纯视觉风格。
  - **原创精品**：如极简墨色、复古杂志、黑金舞台、手账便签、暖白秩序、蓝灰简报、冷感蓝图、森绿编辑等。
  - **m2w 系列**：30 套从 markdown2wechat 移植的主题，前缀 `m2w-`，涵盖多种色系（极客黑、玫瑰紫、灵动蓝、柔和绿等），用户可点名使用（如"用 m2w-geek-black 主题"）。

选择步骤：
1. 用户点名了主题就用它
2. 优先判断是否落入题材型主题（技术长文→原生蓝图，计划/复盘/笔记→青格笔记）
3. 落不进去时，从原创精品风格型主题里挑气质最贴的那套（正式简报选冷静克制的，轻松清单选手账/便签感的，需要仪式感选黑金，长文慢读选极简墨色）
4. m2w 系列主题作为补充选项，当用户明确想要特定色系或风格时推荐

新增主题：在 `references/themes/` 下加一份 `theme-<名字>.md`（沿用现有结构：首行一句话说明适用场景或视觉定位，再给设计令牌、页面外壳、各组件片段），并在 `examples/` 放一篇同名范例，本技能会自动支持，无需改本文件。

## 元素映射要点

主题文件已给出每个组件的完整片段，套用时注意这几条容易出错的地方：

- **严格内联**：只用 `style="..."`，绝不引入 `<style>`、`class`、外部 CSS。这是公众号/邮件能正常显示的前提。
- **代码块和流程图**：不要用 `<pre>`/`<code>`，它们的空白折叠在公众号里不可靠。改用主题的"代码块"组件，内部用 `&nbsp;` 保留缩进、`<br>` 换行。转义交给脚本，别手工拼：
  ```bash
  python scripts/escape_code.py --in /tmp/snippet.txt
  # 或： cat snippet.txt | python scripts/escape_code.py
  ```
  它会 HTML 转义 `< > &` 并把空格/换行转成 `&nbsp;`/`<br>`，输出直接替换组件里的 `{{escaped_code_with_br_and_nbsp}}`。
- **行内代码、加粗强调**：主题若提供了"行内代码""强调短语"这类组件就用它，走内联 `<span>`/`<strong>`；没有就退化成普通文本加内联样式。
- **列表**：Markdown 的 `-`/`1.` 列表用主题里承载列表的组件（名字各异，如"多项列表""节奏条目""清单"等，以主题文件为准），每项一个 `<p>`，自己加 `•` 或序号前缀——不要用 `<ul>/<ol>`，公众号会重置它们的边距。
- **语义化组件**：每套主题都带几个额外组件，名字和数量因主题而异（概念卡片、关键判断、复盘卡片、舞台重点、暖色摘要、轻提示、参考资料组……）。先读主题文件看它有哪些，再按语义对应：Markdown 里出现"核心结论/重点提示/关键判断"就套该主题的重点类组件，"参考链接/延伸阅读"套参考类组件，比用普通段落或引用观感好得多。判断要克制，别把每段都包成卡片。主题没有的组件就不用，退回普通段落。
- **图片**：`![alt](src)` 用"封面图/文内图片"组件；紧跟其后的斜体或说明文字用"图片说明"组件居中显示。图片地址原样保留。
- **表格**：用主题的"表格"组件（若该主题定义了）。表头行套 `thead`，数据行套 `tbody`，注意主题里隔行变色的写法；主题没给表格组件时，按其设计令牌的边框/表面色自拟一个内联样式的 `<table>`。
- **中文排版**：保留中文标点，正文里中英文之间不强行加空格；标题、强调按主题既定字号字重，不要自创样式。

## 输出

默认在 Markdown 同目录（或当前目录）写出同名 `.html` 文件，文件名跟随文章主题或源文件名。完整结构以主题文件的"页面外壳"为准——必须包含 `<!doctype html>`、`<meta charset="utf-8">` 和 viewport，正文在那个 680px 居中容器里。

写完后跑 `python scripts/validate_html.py <文件>` 自检；它会报出 `<style>`、`class`、缺 charset 等会导致粘贴掉样式的问题。最后告诉用户：生成的文件路径、用了哪个主题、为什么。
